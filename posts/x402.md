### TL;DR

- **기계 간 결제 프로토콜:** x402는 AI 에이전트가 API 요청당 비용을 온체인(stablecoin)으로 자동 결제할 수 있게 하는 HTTP 기반 결제 표준입니다. HTTP 402(“Payment Required”) 상태 코드를 활용하여 결제 없이 요청이 오면 결제 정보를 응답으로 보내, 기존의 API 키나 구독 없이 결제가 가능하도록 합니다.
    
- **스테이블코인 & 블록체인:** 결제는 주로 USDC 같은 스테이블코인과 블록체인(예: Base 레이어2)에서 이뤄집니다. 트랜잭션은 약 200ms 정도면 최종 확정되며, 기존 결제의 취소(차지백)가 원천 차단됩니다.
    
- **마이크로페이먼트 지원:** L2 롤업 덕분에 수수료가 극히 낮아져, API 1건당 $0.001 단위의 미세한 과금이 가능합니다. 이를 통해 AI 추론이나 데이터 조회 같은 새로운 페이퍼유즈 모델이 현실화됩니다.
    
- **간단한 요청 흐름:** 에이전트가 API를 호출하면, 결제가 없을 경우 서버는 HTTP 402와 함께 비용 정보(JSON)를 응답합니다. 에이전트는 자신의 암호화 지갑으로 서명된 결제 승인 정보를 재요청에 포함시켜 보내면, 서버는 서명 검증 후 온체인 전송을 수행하고 결과를 반환합니다.
    
- **표준화된 JSON & 서명:** 402 응답에는 `maxAmountRequired`, `assetAddress`, `payTo`(받는 지갑), `network`, `expiresAt`, `nonce`, `paymentId` 같은 항목이 포함됩니다. 클라이언트는 이 정보를 기반으로 EIP-712 방식의 서명을 생성해 결제를 승인합니다.
    
- **체인 및 토큰 무관:** USDC를 예로 들지만, x402는 어떤 ERC-20 스테이블코인이나 블록체인 네트워크도 지원하도록 설계되었습니다. 다양한 정산 방법(온체인, L2 롤업, 채널, 배치 등)을 제공하여 확장성과 비용 효율성을 높입니다.
    
- **레거시 마찰 제거:** 차지백(결제 취소)과 PCI 컴플라이언스, 제3자 승인 없이 즉시 실시간 결제가 가능하여, AI 에이전트나 사용자가 신용카드 없이 곧바로 유료 리소스에 접근할 수 있습니다.

![image](https://raw.githubusercontent.com/tain030/blog-post/main/images/x402.svg)

x402 백서에서는 **현재 결제 시스템이 주로 사람 중심**으로 설계되었음을 강조합니다. 기존 레거시 결제 수단은 신용카드, 구독, 수기 인보이스 등에 의존하며, 결제 승인이 며칠씩 걸리고 높은 수수료와 차지백 위험을 수반합니다. 이러한 제약은 _머신 대 머신_ 결제와 AI 에이전트의 자율적 활동에 큰 마찰로 작용합니다. 예를 들어 AI 모델이 실시간 데이터 API를 사용하려 해도, 계정 등록이나 키 세팅 같은 인간적 과정이 필요하면 자동화가 어렵습니다.

이 때문에 AI 에이전트는 언제든지 즉시 API나 데이터를 구매할 수 있어야 합니다. x402의 동기는 **완전 자율 AI 시스템**을 가능하게 하는 것입니다. 머신러닝 에이전트가 데이터 피드나 클라우드 자원을 스스로 사고 사용할 수 있다면(결제 마찰 없이) 더 효율적이고 혁신적인 서비스를 구축할 수 있습니다. 백서에서는 x402가 *"기계 기반 결제(machine-native payments)"*를 실현하여 API 공급자와 AI 에이전트 간의 신뢰할 수 있는 실시간 거래를 가능케 한다고 설명합니다. 요약하면, AI 주도 경제에서 기존 결제의 비효율(지연, 비용, 제한)이 큰 장애물이므로, 이를 해결해 마찰 없는 디지털 경제를 열자는 것이 배경입니다.

## 개요

x402의 핵심 아이디어는 **HTTP를 통한 자동결제 프로토콜**을 표준화하는 것입니다. 구체적으로, HTTP `402 Payment Required` 상태 코드를 사용하여 AI 에이전트가 각 요청마다 온체인 화폐(스테이블코인)로 결제할 수 있게 합니다. 즉, 결제 없이 요청하면 서버가 `402`를 응답하고 JSON으로 결제 정보를 줍니다. 이 JSON에는 요청당 비용, 사용할 토큰 주소, 받아야 할 지갑 주소, 네트워크 정보 등이 담깁니다.

클라이언트(에이전트)는 이 정보를 바탕으로 암호화 지갑에서 해당 금액만큼의 토큰 전송에 대한 **서명된 승인**을 생성하여, 이 승인 문구를 포함해 요청을 재전송합니다. 서버는 받은 서명 데이터를 검증하고, 실제 온체인 거래를 발행하여 토큰을 전송한 뒤(결제 완료), 원래 요청에 대한 응답을 반환합니다.

이 흐름을 통해 x402는 API 키나 사용자 계정 없이도 요청당 결제를 가능하게 합니다. 백서 예시대로 한 줄의 미들웨어 코드(`paymentMiddleware(amount: "0.10", address: "0x...")`)를 추가하기만 하면 해당 API가 과금됩니다. 요약하면, x402는 **기존 HTTP 인프라에 블록체인 결제를 가볍게 통합**하여, AI 에이전트가 각 호출마다 자율적으로 결제할 수 있는 환경을 만드는 것이 핵심 개념입니다.

## 기술 구조 및 모델

x402 프로토콜은 크게 세 부분으로 구성됩니다: **AI 에이전트(고객)**, **웹 서버(서비스 제공자)**, 그리고 **블록체인 네트워크**. 구현 예로, 서비스 제공자는 NodeJS/Express 같은 서버에 x402 미들웨어를 붙입니다. 이 미들웨어는 엔드포인트의 가격 설정(예: 0.10 USDC)과 개발자 지갑 주소를 포함한 구성으로 초기화합니다.

1. **클라이언트 요청:** AI 에이전트나 애플리케이션이 API 엔드포인트에 접근을 요청합니다.
    
2. **서버의 402 응답:** 서버는 요청에 결제 정보가 첨부되지 않았음을 감지하면 HTTP 402 상태와 함께 JSON을 반환합니다. 예를 들어:
    

3. `{   "maxAmountRequired": "0.10",   "resource": "/api/market-data",   "description": "Access to data requires payment.",   "payTo": "0xABCDEF...1234",   "asset": "0xA0b8...6048",   "network": "ethereum-mainnet" }`
    
    이 JSON은 요구되는 결제 조건(금액, 자산, 네트워크 등)을 명확히 전달합니다.
    
4. **결제 서명 생성:** 에이전트는 받은 JSON을 확인하고, 자신의 지갑에서 **EIP-712** 방식의 서명을 만듭니다. 서명에는 응답의 모든 필드, 실제 결제 금액(≤ `maxAmountRequired`), 그리고 현재 타임스탬프가 포함됩니다. 이는 지갑에서 “이 API 요청을 위해 이만큼을 전송”한다고 명시하는 암호화 서명이 됩니다.
    
5. **재요청 및 검증:** 에이전트는 이 서명 메시지를 포함해 원래 요청을 다시 전송합니다. 서버의 x402 미들웨어는 서명의 유효성을 검증하고, 아직 유효 기간(`expiresAt`)이 남아있는지, 서명된 금액이 최대치를 넘지 않는지 확인합니다.
    
6. **블록체인 전송:** 검증이 완료되면 서버는 해당 토큰을 실제로 전송하기 위해 블록체인 트랜잭션을 발행합니다. 예를 들어, Base 체인에서 USDC 전송 트랜잭션을 보내면 됩니다.
    
7. **결과 반환:** 토큰 전송 트랜잭션이 컨펌(최종 확정)되면, 서버는 클라이언트에게 API 요청 결과를 반환합니다. 즉시 데이터를 응답해줍니다.

기술 사양에서 언급된 결제 요청 형식과 서명 메시지 요구사항 덕분에 이 모든 과정이 엄격히 표준화됩니다. 또한, x402는 온체인 결제 외에도 **Layer-2 롤업, 페이먼트 채널, 배치 전송** 같은 다양한 정산 방법을 지원하여 거래량이나 빈도에 맞추어 효율성을 높일 수 있습니다. 기본적으로 x402 아키텍처는 기존 웹 API에 최소한의 변경만을 요구하며, 백서 예시처럼 단 한 줄의 코드 추가로 “결제가 필요한 엔드포인트”를 만들 수 있습니다.

## 핵심 메커니즘

- **HTTP 402 + 표준 JSON 요청:** x402의 중심은 HTTP 402 상태 코드를 사용하여 결제를 요청하는 것입니다. 미결제 상태에서 API를 호출하면 서버가 `402 Payment Required` 응답을 보내고, 결제 세부 정보를 JSON으로 전달합니다. 이 JSON에는 요구 금액, 토큰 주소, 수취 주소, 네트워크, 만료 시간, nonce, paymentId 등 결제 인보이스 전체가 담깁니다. 이로써 에이전트는 자동으로 결제 처리를 시작할 수 있습니다.
    
- **결제 요청 필드 (Payment Request Format):** 402 응답의 JSON 필드에는 다음이 포함됩니다: `maxAmountRequired` (최대 결제 금액), `assetAddress` (결제 토큰 계약), `paymentAddress`/`payTo` (지갑 주소), `network` (블록체인 네트워크), `expiresAt` (만료 타임스탬프), `nonce`, `paymentId` 등. `nonce`와 `paymentId`는 같은 요청이 중복되거나 재사용되는 것을 막아 보안을 강화합니다. 만료 시간과 금액 한계로 요청이 특정 조건에서만 유효하도록 제한합니다.
    
- **EIP-712 서명 (Payment Authorization):** 에이전트는 클라이언트 지갑을 사용해 받은 결제 요청을 EIP-712 포맷으로 서명합니다. 이 서명 메시지에는 요청의 모든 필드와 실제 보낼 금액, 그리고 생성 시각이 들어갑니다. EIP-712는 지갑 UI에서 읽기 쉽도록 메시지를 구조화하며, 서명을 통해 해당 요청에 대한 결제 권한이 암호화되어 증명됩니다. 서버는 이 서명을 확인하여 해당 요청에 대해 지갑 소유자가 승인을 했음을 검증합니다.
    
- **온체인 정산 (Transaction Settlement):** 결제가 유효하면 서버는 실제로 토큰 전송 트랜잭션을 발행합니다. 백서에서 제시한 정산 방법에는 여러 옵션이 있습니다:
    
    - _온체인 전송:_ 기본 방식으로, Ethereum 계열 체인에 직접 트랜잭션을 보냅니다.
        
    - _레이어2 롤업:_ (예: Base) 같은 빠른 네트워크를 활용하면 수수료를 낮추고 속도를 높일 수 있습니다.
        
    - _페이먼트 채널:_ 자주 거래하는 파트너 간에 오프체인 채널을 사용하여 처리 부담을 줄입니다.
        
    - _배치 송금:_ 여러 번의 소액 결제를 묶어서 한 번의 트랜잭션으로 처리할 수도 있습니다.  
        이 유연성 덕분에 고빈도 수백·수천 건의 마이크로결제도 효율적으로 처리할 수 있습니다.
        
- **즉시 확정 및 보안:** 블록체인을 사용하기 때문에 결제는 **즉시 확정(finality)**됩니다. 백서에 따르면 Base L2에서는 대략 200ms 내에 최종 확정되어, API 제공자는 자금이 확정된 즉시 서비스를 제공할 수 있습니다. 또한, 승인된 트랜잭션은 되돌릴 수 없어 **차지백 위험이 없습니다**. `nonce`와 `paymentId`로 각 요청을 고유하게 묶어두기 때문에, 한 번 사용된 결제 메시지는 재사용할 수 없습니다. 에이전트가 갖고 있는 개인키만이 해당 서명을 만들 수 있으므로, 결제 서명이 유효하면 그 거래는 실제 지갑 소유자가 승인한 것으로 간주됩니다.
    
- **간단한 통합:** 개발자는 종종 다음과 같은 코드 한 줄만 추가하면 됩니다:

- `app.get('/premium', x402PaymentRequired({   amount: "0.10",   address: "0x1234...",   assetAddress: "0x2345...", // USDC 컨트랙트   network: "base-mainnet" }), (req, res) => {   res.json({ data: "Premium content" }); });`
    
    이처럼 x402 미들웨어가 붙은 엔드포인트는 유효한 결제가 확인될 때만 응답을 반환합니다.

## Security / Correctness / Guarantees

x402의 보안성은 온체인 결제의 특성을 크게 활용합니다:

- **불가역적 결제:** x402 결제는 블록체인에서 이루어지므로, 한 번 확정되면 되돌릴 수 없습니다. 백서에서도 확인하듯, x402 트랜잭션은 매우 빠르게 최종 확정되어「충전 환불 창(rolling chargeback windows)이 없음」을 강조합니다. 이는 전통적 결제(신용카드/PayPal)에서 볼 수 있는 며칠짜리 결제 처리나 몇 달간의 반환 위험이 존재하지 않음을 의미합니다.
    
- **암호화된 서명:** 각 결제는 EIP-712로 서명됩니다. 이 서명은 API 호출에 정확한 금액과 모든 파라미터를 명시적으로 포함하므로, 에이전트가 승인한 내용 그 외의 어떤 것도 실행될 수 없습니다. 다시 말해, 에이전트가 지정한 범위를 벗어난 과금이나 다른 요청 결제 시도가 서명 검증에서 걸러집니다. 개인키가 있어야만 서명을 생성할 수 있기 때문에, 서명이 유효하다면 진짜 지갑 소유자만 그 결제를 승인했음을 보장합니다.
    
- **재사용 방지:** JSON 페이로드에 포함된 `nonce`와 `paymentId`는 각 결제 요청을 고유하게 식별합니다. 같은 메시지를 두 번 쓰는 것을 막아 **재생 공격**을 방지합니다. 또한, 요청에 만료 시간(`expiresAt`)을 넣어두어 일정 시간이 지나면 해당 결제는 자동으로 무효화됩니다.
    
- **블록체인 신뢰:** x402의 신뢰성은 기반으로 삼은 블록체인 인프라에 의존합니다. 즉, Base나 Ethereum 같은 퍼블릭 체인이 안전하게 운영되고, USDC 등의 스테이블코인이 제 기능을 한다면, x402 결제도 안전합니다. 백서에서 언급한 대로 “permissionless blockchain infrastructure”를 쓰므로, 중앙화된 검열이나 실패 위험이 줄어듭니다. 요약하자면, x402는 기본적으로 블록체인의 영구성(finality)과 암호 서명 보안을 사용하여 결제의 정확성과 보안을 보장합니다.

## 기존 기술과의 비교

x402는 주로 기존 지불 시스템(신용카드, 페이팔 등)과 명확히 차별화됩니다. 백서에서 제시한 표와 시나리오를 요약하면:

- **전통적인 결제 수단 대비:** 신용카드나 페이팔은 수수료가 상당히 높고, 처리 시간이 길며, 취소 위험을 안고 있습니다. 예를 들어 신용카드는 거래당 0.30달러+2.9%의 수수료에 며칠간의 결제 처리가 필요하고 120일 이내 취소가 가능할 수 있습니다. 반면, x402는 거의 제로에 가까운 수수료(“Free”로 표기)로 즉시 결제 확정이 가능하며, 차지백(취소) 위험이 전혀 없습니다. PayPal도 실시간 승인 후 며칠에 걸쳐 결제되는 구조로 취소 가능성이 있지만, x402는 곧바로 최종 확정됩니다. 핵심적으로, x402는 수수료, 대기시간, 취소 리스크를 모두 제거함으로써 AI 중심의 마이크로결제에 최적화되어 있습니다.
    
- **암호화폐 결제 비교:** 일부 암호결제 솔루션(예: Stripe Crypto)나 직접 이더리움 사용과도 대비됩니다. Stripe의 암호화폐 결제 역시 내부적으로 블록체인에 의존하며, 전송 후에야 결제 확정되고 취소 불가 여부는 네트워크에 따라 다릅니다. 이더리움 메인넷(Ethereum L1)은 수수료가 1–5달러 수준으로 높고 확정까지 1–2분이 걸립니다. 반면 x402는 낮은 비용의 L2를 활용해 수수료를 거의 $0로 낮추고(거의 $0.0001 수준) 확정 속도를 0.2초로 단축했습니다. TPS(초당 처리량)에서도 이더리움 L1은 약 15–20 TPS인 반면, Base 상의 x402는 수백~수천 TPS를 지원합니다. 따라서 대규모 다중 요청 처리 면에서 x402가 훨씬 우월합니다.
    
- **에이전트 결제 흐름:** 백서의 예시 시나리오를 보면, 전통 방식에서는 AI 에이전트가 데이터 API를 쓰려면 여러 개의 계정을 만든 뒤 API 키를 받아야 합니다. 하지만 x402 모델에서는 에이전트가 API를 요청하기만 하면 402로 결제 요청이 응답되고, 에이전트는 USDC를 첨부해 재요청하면 바로 사용권이 주어집니다. 즉, 사전 승인이나 계정 절차 없이 즉각 결제-사용이 가능합니다. 이처럼 기존 방식에서는 불가피했던 여러 단계가 x402에선 한 번의 API 호출로 대체됩니다.
    
- **기존 표준과의 차이:** 현재 널리 사용되는 결제 표준(예: 신용카드 결제 게이트웨이 API, OAuth 등)에는 기계간 자동결제 같은 개념이 없습니다. x402는 HTTP 자체를 확장해 결제 상태를 전달한다는 점에서 독창적입니다. 이는 기존 기술과 직접적인 경쟁 대상이 아니라, 새로운 계층의 프로토콜로 볼 수 있습니다.

종합하면, x402는 기존의 모든 결제 구조를 다 대체하기보다는, AI/에이전트 중심의 새로운 결제 수요를 위한 **전용 툴킷**을 제공합니다. 주요 차이점은 “미리 계정 만들 필요 없이, 요청할 때 결제”라는 점과, **완전한 블록체인 결제의 이점(즉시 확정, 낮은 수수료)**을 활용한다는 것입니다.

## 제한점 및 향후 과제

백서 자체는 제한사항을 명시적으로 열거하지 않지만, 다음과 같은 고려사항과 미해결 이슈가 있습니다:

- **에이전트와 지갑 필요:** x402는 에이전트가 암호화 지갑을 보유하고 있어야 한다는 전제를 깔고 있습니다. 즉, USD를 USDC로 환전하거나 지갑을 설정하는 방식 등, 실제 온보딩 과정에 대한 설명은 없습니다. 이 과정은 실제 도입 시 장애가 될 수 있습니다.
    
- **블록체인 의존성:** L2 사용으로 비용과 속도가 개선됐지만, 여전히 네트워크 혼잡이나 관리 문제에 따라 성능이 달라질 수 있습니다. 선택한 블록체인이 문제를 겪을 때의 대비책은 제시되지 않았습니다.
    
- **규제 및 컴플라이언스:** x402는 PCI를 피할 수 있지만, 자체적으로는 AML/KYC와 같은 이슈가 어떻게 될지 언급이 없습니다. 예를 들어, 어떤 기관이 결제 받은 토큰 처리에 대한 법적 요구를 어떻게 준수할지 등이 불분명합니다.
    
- **스테이블코인 안정성:** 스테이블코인이 항상 1:1 페그를 유지할 것인지 가정합니다. 만약 스테이블코인의 가치가 변동하거나 규제 문제가 발생하면 어떻게 될지 논의가 없습니다.
    
- **초소액 결제 관리:** 개별 거래 단가는 극소액이지만, 이를 관리하고 회계 처리하는 방식(배치 정산, 회계 시스템 연동 등)은 설계에서 다루지 않습니다.
    
- **보안 가정:** 기본적으로 블록체인의 보안과 서명 키 보안에 의존합니다. 지갑이 탈취되면 결제를 통제할 방법이 없는데, 키 관리 방안은 백서에 없습니다.
    

결국 이 프로토콜은 “기계가 직접 결제할 수 있게 해준다”는 비전을 중심으로 제안되었지만, 실운영 환경에서 맞닥뜨릴 수 있는 법적·운영적 도전 과제들은 더 연구가 필요합니다.

## 실용적 함의

x402가 실제로 도입된다면 다음과 같은 변화가 예상됩니다:

- **마이크로페이먼트 활성화:** 가장 큰 효과는 지금까지 불가능했던 초소액 결제가 가능해진다는 점입니다. 백서에서 언급한 것처럼, $0.001 수준의 결제도 수익성을 보장할 수 있어 개발자들은 API 또는 콘텐츠 사용량에 따라 자유롭게 요금을 매길 수 있게 됩니다. 예를 들어, AI 모델의 추론 요청당 소액 과금이나 실시간 데이터 스트림 구독 등에 쓰일 수 있습니다. 이는 기존의 구독·패키지 모델로는 실현하기 어려웠던 비즈니스 모델을 열어줍니다.
    
- **결제 프로세스 단순화:** 서비스 제공자는 복잡한 청구 시스템을 구축할 필요가 없습니다. x402를 적용하면 API 호출만으로 결제가 관리되기 때문에 인보이스, 결제 오류, 사기 방지 등 여러 운영 비용이 줄어듭니다. 백서에 따르면 이를 통해 기업은 “높은 수익, 낮은 비용, 그리고 원활한 결제 경험”을 얻을 수 있습니다. 소비자나 에이전트도 별도의 계정이나 선불 포인트 없이 즉시 구매할 수 있어 UX가 크게 개선됩니다.
    
- **글로벌 접근성:** 퍼블릭 블록체인을 사용하므로, 은행 계좌나 신용카드가 없는 지역에서도 기술만 있으면 서비스를 이용할 수 있습니다. 이는 은행 인프라가 부족한 개발도상국 사용자에게도 유료 API나 콘텐츠를 개방하는 것을 가능하게 합니다. 백서에서 언급한 대로, 전 세계에 별도의 금융 인프라 없이 접근할 수 있게 됩니다.
    
- **AI 에이전트 경제 활성화:** AI 시스템 개발자 입장에선 x402로 에이전트가 필요한 자원을 스스로 구매하도록 만들어, 워크플로우가 자동화됩니다. 가령, 연구용 AI가 필요할 때마다 데이터베이스 API를 호출해 데이터를 사오거나, 추가 연산이 필요하면 클라우드 인스턴스 사용 시간을 구매할 수 있습니다. 인간의 개입 없이도 AI가 스스로 자기 개발에 필요한 리소스를 조달할 수 있다는 점에서 에이전트 경제가 발전할 수 있습니다. 백서도 이로써 “AI 중심의 상거래를 가속화”한다고 강조합니다.
    
- **개발자 생태계:** 표준화된 프로토콜이기 때문에, 지갑 제공자와 API 제공자 사이에 공통 인터페이스가 생깁니다. 예를 들어, 다양한 서비스들이 x402를 지원하면, 한 번 지갑 연동을 한 AI 에이전트는 여러 서비스에 쉽게 결제할 수 있게 됩니다. 이는 점차 블록체인 결제 방식을 웹 표준의 한 축으로 키우는 효과가 있을 수 있습니다.
    

초기 사용 사례로는 AI 모델에게 실시간 시장 데이터를 판매하는 API, 온디맨드 계산 자원 제공, 혹은 기사·문서 단위 과금 뉴스 서비스 등이 유력합니다. 백서 예시처럼 신문 기사 한 건 구입, 지불벽(paywall)을 위한 기사 단위 결제가 이미 가능한 모습을 보여줍니다. 요약하면, x402는 **AI 서비스와 디지털 콘텐츠 산업 전반의 과금 방식을 혁신**할 잠재력이 있습니다.

## 요약

Coinbase 개발팀의 x402 백서는 **AI 에이전트와 웹 서비스가 온체인 화폐로 자동 결제할 수 있는 HTTP 프로토콜**을 제안합니다. HTTP 402 상태 코드와 표준화된 JSON을 활용하여 각 API 호출에 대금 지불이 되도록 설계되었습니다. 에이전트는 자신의 지갑으로 결제 승인을 EIP-712 서명 방식으로 생성해 서버에 제출하면, 서버는 이 서명을 검증하고 즉시 블록체인에서 결제를 완료합니다. 그 결과, 기존의 API 키나 구독 모델 없이도 즉시 결제, 즉시 서비스가 가능해집니다. 이 프로토콜은 거의 제로에 가까운 수수료와 즉시 확정 결제를 제공해, 전통 결제의 번거로움을 제거합니다. 게다가 단일 코드 추가만으로 구현 가능하므로 개발자에게 통합 비용이 적습니다. 종합하면, x402는 AI 주도의 상거래를 위해 웹과 블록체인을 매끄럽게 연결하는 새로운 표준으로 자리매김할 것으로 기대됩니다.
